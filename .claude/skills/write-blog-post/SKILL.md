# ブログ記事を書く

kkito blog の記事を、調査から公開（PR作成）まで一貫して仕上げるためのスキル。

**このスキルが扱うのは執筆のプロセスと作法**です。フォルダ構成・frontmatterの項目・使えるMarkdown記法といった機械的な規則は [docs/WRITING_GUIDE.md](../../../docs/WRITING_GUIDE.md) にまとまっているので、記事ファイルを作る前に必ず一度読むこと。ここではそれを前提に、「どう調べ、どう構成し、どう検証するか」を扱う。

## 全体の流れ

```
1. 何を書くか固める      ← ユーザーの意図を掴む
2. 事実を確認する        ← 最重要。推測で書かない
3. 構成を組む
4. 書く
5. 検証する              ← ビルド + 表示確認
6. コミットしてPRを作る
```

1と3を先に固めてから2に入ると速い。何を書くか決まらないまま調べ始めると、記事に使わない情報まで集めることになる。

## 1. 何を書くか固める

まずユーザーが何を伝えたいのかを掴む。記事のテーマだけ渡されることが多いので、次を確認する。

- **読者は誰か** — 同じ技術を使う人か、その分野を知らない人も含むか
- **一番伝えたいことは何か** — これが冒頭に来る。複数あるなら優先順位を聞く
- **どこまで踏み込むか** — 使い方の紹介か、設計判断の話か、失敗談も含むか

ユーザーが「〇〇について書いて」とだけ言った場合、勝手に範囲を決めずに、書き出す前に「こういう構成で考えていますが、入れたい話はありますか」と一度確認したほうが早い。書き上げてから方向性が違うと分かるのが一番もったいない。

## 2. 事実を確認する（最重要）

**推測で書かない。** このスキルで一番大事なのはここ。

技術記事は具体的であるほど価値が出るが、具体的な記述は間違っていると価値がマイナスになる。特に**設計の意図や動機は、コードを読んでも分からない**。

### 何を調べるか

対象がリポジトリやツールなら、書く前に実物を見る。

- README（日本語版があればそちらも）
- ディレクトリ構成、主要なモジュールの分かれ方
- 設定ファイル（package.json、ワークフロー、devcontainer.json など）
- 公開ページがあれば実際に開く
- APIやコマンドの実際のシグネチャ

コード例を載せるなら、**実際のAPIと一致させる**。それらしい引数名をでっち上げない。

### 何をユーザーに聞くか

**「なぜそう作ったか」は本人にしか分からない。** ここを推測で埋めると高確率で外れる。

たとえば「Goで単一バイナリのCLIを切り出した」という事実からは、「Python環境がない人向けだろう」とも「一括処理を速くしたかったのだろう」とも読める。前者だと決めつけて書いて、実際は後者だった、ということが実際に起きた。

だから設計判断の話を書くときは、事実（何をしたか）と意図（なぜしたか）を分けて、意図の部分はユーザーに確認する。

### 分からないことは書かない

調べても確証が持てないことは、書かないか、断定を避ける。「〜のはずです」と書くくらいなら、その話題自体を落としたほうが記事は強くなる。

### どこまで調べれば十分か

「推測で書かない」は「すべてを調べ尽くす」という意味ではない。**記事に書く内容だけを裏取りすればいい。**

判断の目安はこうする。

- **書こうとしている文が具体的な事実を含むか** — 含むなら裏を取る。含まないなら不要
- **その事実が間違っていたら記事の価値が下がるか** — 下がるなら裏を取る

たとえば「Dev Containerについて」という依頼で、記事の軸を「1つの設定ファイルの変遷」に決めたなら、調べるのはその1リポジトリの設定と履歴でいい。関連しそうな5リポジトリすべてを精査するのは、記事に書かない情報まで集めていることになる。

**調査は構成を決めてから始めるほうが早い。** 何を書くか決まっていない状態で調べると、際限なく広がる。先に「この3つの節を書く」と決めて、その3つに必要な事実だけを取りに行く。

逆に、調査の途中で「この話のほうが面白い」と気づいたら構成を変えてよい。その場合も、変えた構成に必要な範囲だけを追加で調べる。

## 3. 構成を組む

### 冒頭で趣旨を示す

読者は冒頭数行で読むかどうかを決める。**この記事で何が分かるのかを最初に置く**。

結論を先に出す構成が効く。「先に結論から。手順はこれだけです」と書いて箇条書きを2行出す、といった形。

### 見出しは `##` から

`#`（h1）はページ側でタイトルとして表示されるので本文では使わない。`##` と `###` が目次に自動で拾われる。

見出しは内容を予告するものにする。「設計思想」より「設計思想：作業の性質に道具を合わせる」のほうが、目次を見ただけで話の中身が伝わる。

### 他の記事へリンクする

記事間のリンクは**相対パス**で書く。ルート相対で書くとビルドが失敗する。

```markdown
✅ [前回の記事](../pymea-mea-analysis/)
❌ [前回の記事](/posts/pymea-mea-analysis/)
```

サイトがGitHub Pagesのサブパスで配信されるため、`/` から書いたリンクには `base` が付かず本番で404になる。ローカルでは正常に見えてしまうので、ビルド時に弾くようにしてある。

### 図・表・コードを使い分ける

- **Mermaid図** — 処理の流れ、構成の階層。文章で追いにくい関係を示すのに向く
- **表** — 対応関係の列挙（技術スタック、オブジェクトと役割、比較）
- **コードブロック** — 実際の使い方。言語指定を忘れない

図はビルド時にSVG化されるので、執筆時は ```mermaid のコードブロックを書くだけでよい。

### 課題も書く

うまくいった話だけだと薄くなる。**残っている課題や、まだ解けていない問題を正直に書く**と記事に厚みが出る。「Dev Containerは動作が重い。WSL側に置けば改善するはずだが、それをどう伝えるかが難しい」といった書き方。

### 締める

最後に要点を箇条書きで振り返り、それらが何に向かっているのかを一文でまとめると収まりがよい。リポジトリへのリンクと、読者への一言を添える。

## 4. 書く

### 文体

- **です・ます調**
- 一文は短く。読点で延々とつながない
- 強調は要点だけに絞る。太字だらけにすると強調が効かなくなる
- **自賛のトーンを避ける** — 「面白いのは〜していることです」より「〜しています」。事実を淡々と書くほうが伝わる
- 専門用語は初出時に一言添える。読者に前提知識を要求しすぎない

### タイトル

- 記事一覧とOGPに出るので、**モバイルで2行に収まる長さ**を目安にする
- 要素を詰め込みすぎない。3つ並べるなら2つに削れないか考える
- 要素を並べるときは区切り記号を入れる。スペースだけで繋ぐと一続きに読まれる

### description

一覧・OGP・検索結果に出る。記事の趣旨を1〜2文で。タイトルの繰り返しにせず、補う情報を書く。

## 5. 検証する

書き終えたら必ず検証する。**ビルドが通ることと、表示が崩れていないことは別**なので両方見る。

```bash
npm run build
```

frontmatterの不備・画像の欠落・Mermaid構文エラー・ルート相対リンクはここで落ちる。落ちたらメッセージに直し方が書いてあるので従う。

### 太字の崩れを確認する

日本語の括弧と太字の組み合わせは、**ビルドが成功したまま表示だけ壊れる**という厄介な性質がある。書いた記事について毎回確認すること。

```bash
grep -o '\*\*[^*]\{0,30\}' dist/posts/<slug>/index.html
```

何も出なければOK。出た場合、それがインラインコード内のglobパターン（`docs/**/*.md` など）なら正常。本文中なら崩れているので [docs/WRITING_GUIDE.md](../../../docs/WRITING_GUIDE.md) の「落とし穴」に従って直す。

### 表示を目で見る

Mermaid図・コードブロック・表を含む記事は、実際にレンダリングを見る。ライトとダークの両方を確認する（図はテーマごとに別のSVGが生成される）。

```bash
npm run preview   # localhost:4173/tech-blog-kkito/posts/<slug>/
```

Playwrightが使えるならスクリーンショットを撮って確認するのが確実。

### コード例の正しさ

記事に載せたコードは読者がコピーする前提で見直す。設定ファイルの例なら、その形式として妥当か確認する（JSONに末尾カンマを残さない、など）。

## 6. コミットしてPRを作る

`main` へ直接コミットしない。

```bash
git checkout main && git pull --ff-only
git checkout -b docs/post-<slug>
# 執筆・検証
git add content/posts/<folder>/
git commit -m "docs: 「<タイトル>」の記事を追加"
git push -u origin docs/post-<slug>
gh pr create --base main --title "..." --body "..."
```

PR本文には、記事の構成の要点と、検証した内容（ビルド・表示確認）を書く。詳細は [docs/CONTRIBUTING.md](../../../docs/CONTRIBUTING.md)。

マージすると GitHub Actions が走り、数分で公開される。

## 加筆・修正の場合

既存記事に手を入れるときも流れは同じだが、次に注意する。

- **ユーザーが自分で編集していることがある** — 作業前に `git diff` で現状を確認する。エディタで開いたまま編集が競合すると、直したはずの箇所が巻き戻ることがある
- **一部を直したら全体の整合性を見る** — ある節の主張を変えたら、冒頭の趣旨や「おわりに」の要約とズレていないか確認する
- 節を削除したら、目次以外の場所（ディレクトリツリーの注記など）に残っていないか `grep` で確認する

## よくある失敗

| 失敗 | 対策 |
| --- | --- |
| 設計の意図を推測で書いて外す | 事実と意図を分け、意図はユーザーに確認する |
| コード例の引数名がでっち上げ | 実際のREADMEやソースからコピーする |
| 太字が崩れたまま公開 | ビルド後に `grep` で確認する |
| タイトルが長くモバイルで3行 | 要素を2つまでに絞る |
| 記事を書き上げてから方向性の相違が判明 | 書き出す前に構成を一度見せる |
| 「〜のはずです」という曖昧な記述 | 確証がなければその話題を落とす |
| 記事に書かない範囲まで調べて時間を使う | 構成を決めてから、書く内容の裏取りだけする |
| 記事間リンクを `/posts/...` で書く | 相対パス `../<slug>/` で書く（ビルドで弾かれる） |
