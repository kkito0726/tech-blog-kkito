# 記事執筆ガイド

記事を追加するたびに参照するドキュメント。ディレクトリ構成・frontmatter・Markdown記法・画像の置き方・公開までの流れをまとめる。

## 1. 記事の追加場所（1記事1フォルダ）

記事は `content/posts/` 配下に **1記事1フォルダ** で置く。フォルダ名は次の形式に固定する。

```
content/posts/YYYY-MM-DD-<slug>/
├── index.md       ← 本文（必須・このファイル名で固定）
├── some-image.png ← 画像（任意・本文から相対パスで参照）
└── diagram.svg
```

- **`YYYY-MM-DD`**: 公開日の目安（並び順には使われない。並び順はfrontmatターの`date`が基準）。フォルダ名の先頭がこの形式でない場合、次のスラッグ抽出が効かず、日付部分を含んだ文字列がそのままURLになってしまう
- **`<slug>`**: 記事のURLになる部分。**英数字とハイフンのケバブケース推奨**（例: `vite-ssg-setup`）。日本語や記号を使うことも技術的には可能だが、URLがエンコードされて読みにくくなるため非推奨
- 本文ファイル名は必ず **`index.md`**。他のファイル名では記事として認識されない

### URLとの対応

フォルダ名から日付プレフィックス（`YYYY-MM-DD-`）を取り除いた部分が、そのままURLのスラッグになる。

| フォルダ名 | 公開URL |
|---|---|
| `content/posts/2026-07-14-vite-ssg-setup/` | `/posts/vite-ssg-setup/` |

**同じスラッグ（日付を除いた部分）を持つフォルダが2つ以上あるとビルドエラーになる。** 過去記事と同じテーマで書く場合もスラッグは必ず変える。

## 2. frontmatterの書き方

`index.md` の先頭に `---` で囲んだYAMLを書く。

```markdown
---
title: 記事のタイトル
date: 2026-07-14
description: 一覧・OGP・検索結果に使われる短い説明文
draft: false
---

ここから本文
```

| 項目 | 必須 | 型 | 説明 |
|---|---|---|---|
| `title` | ✅必須 | 文字列 | 記事タイトル。空文字は不可 |
| `date` | ✅必須 | `YYYY-MM-DD` | 公開日。**一覧の並び順はこの値が基準**（新しい順）。同じ`date`の記事が複数ある場合はフォルダ名の降順で並ぶ |
| `description` | 任意 | 文字列 | 一覧カード・OGPカード・検索結果に使われる要約文。1〜2文が目安 |
| `draft` | 任意 | `true` / `false`（既定`false`） | `true`にすると本番ビルドで一覧・詳細ページから除外される。**`npm run dev`では表示される**ので、下書きの見た目確認は開発サーバーで行う |

`title`か`date`が欠けている、または`date`が日付として解釈できない場合、**ビルドが失敗する**（意図的な仕様。壊れた記事が公開されるのを防ぐ）。

## 3. 本文の書き方（Markdown）

GFM（GitHub Flavored Markdown）に対応している。使えるもの・使えないものは以下の通り。

### 使えるもの

- 見出し `##`（h2）・`###`（h3） — **`#`（h1）は使わない**。タイトルはページ側で別途大きく表示されるため、本文は`##`から書き始める
- 通常のリスト・番号付きリスト・チェックリスト（`- [ ]`）
- テーブル（`| a | b |` 記法）
- 太字・イタリック・取り消し線（`~~text~~`）
- リンク・画像
- 引用（`>`）
- 水平線（`---`）
- コードブロック（次項）
- 脚注（`[^1]`）
- 裸のURL（自動的にリンクになる）

### 見出しと目次(TOC)の関係

`##`・`###`で書いた見出しは、記事詳細ページの目次に自動的に拾われる。見出しが1つもない記事では目次自体が非表示になる。`####`（h4）以下は目次には出ない。

### 使えないもの

- **生のHTMLタグ**（`<div>`など）はセキュリティ上レンダリングされない。表現したいことがある場合はMarkdown記法で代替するか、コンポーネント追加が必要ならその旨相談する

### 落とし穴: 太字（`**`）と日本語の括弧

太字の内側が `「` や `（` などの括弧で始まる（または終わる）と、**太字にならず `**` がそのまま画面に表示される**ことがある。

```markdown
❌ 文中で**「括弧」の状態**になった
❌ 文中で**「括弧」**の状態になった
   → どちらも「**」が本文にそのまま出てしまう
```

**原因**: Markdown（CommonMark）の仕様。`**` の内側が約物（記号）で、かつ外側が文字だと、強調の開始・終了として解釈されない。**ビルドは成功してしまう**ため、エラーでは気づけない。

#### 確実な書き方

一番安全なのは、**太字の内側を「かな・漢字・英数字」で始めて終える**こと。括弧は太字の外に出す。

```markdown
✅ 文中で「**括弧の中だけ**」の状態になった   ← 括弧を太字の外に出す（推奨）
✅ 文中で**括弧の状態**になった              ← そもそも括弧を含めない
✅ 文中で**括弧「を含む」状態**になった       ← 括弧が内側の途中にあるのはOK
```

括弧ごと太字にしたい場合は、**太字の直前に句読点か行頭を置けば成立する**。

```markdown
✅ 文中で、**「括弧」の状態**になった   ← 直前が読点
✅ 文中です。**「括弧」の状態**になった  ← 直前が句点
❌ 文中で**「括弧」の状態**になった      ← 直前がかな・漢字だと崩れる
```

ただしこの条件は前後の文字に依存して分かりにくいので、**迷ったら括弧を太字の外に出す**書き方に統一するのが安全。

#### 崩れていないか確認する

ビルド後の成果物に `**` が残っていないかで確認できる（コードブロック内の `**` は除く）。

```bash
npm run build
grep -o '\*\*[^*]\{0,30\}' dist/posts/<slug>/index.html
```

## 4. コードブロック（シンタックスハイライト）

三連バッククォートの後に言語名を書くと、ビルド時に自動で色付けされる（Shiki使用、ライト/ダーク両テーマ対応）。

````markdown
```typescript
const greeting: string = 'hello'
```
````

- 対応言語はShikiがサポートする言語全般（`typescript`, `tsx`, `javascript`, `bash`, `yaml`, `json`, `python`, `go`など主要言語は網羅）
- 言語名を省略した場合や存在しない言語名を指定した場合は、色付けなしのプレーン表示にフォールバックする（**ビルドは失敗しない**）
- クライアント側でのハイライト処理は行わない（ビルド時に完結するのでページ表示が速い）

## 5. 画像の追加

画像は記事と同じフォルダに置き、本文からは **相対パス** で参照する。

```markdown
![アーキテクチャ図](./architecture.png)
```

- パスは `./` または `../` で始まる相対パスのみビルド時に検証・変換される。**参照先の画像ファイルが存在しない場合はビルドエラーになる**（パスのtypoに気づける）
- 外部URL（`https://...`）や `/` から始まる絶対パスの画像はそのまま出力される（存在チェックはされない）
- 変換後の画像はハッシュ付きファイル名で配信され、`loading="lazy"` が自動付与される

## 6. 下書きとして書く

書きかけの記事も安心してpush・コミットできる。

```yaml
---
title: (仮) まだ検討中のタイトル
date: 2026-07-20
draft: true
---
```

`draft: true` の記事は本番ビルド（`npm run build`）で一覧・詳細ページの両方から除外される。公開するときは `draft: true` を削除するか `false` に変更する。

## 7. ローカルでの確認方法

```bash
npm run dev
```

開発サーバーでは **下書きも含めて** すべての記事が表示される。見出しの目次・画像・コードハイライトを実際の見た目で確認できる。

本番相当（下書き除外・OGPタグ埋め込み込み）で確認したい場合は次を使う。

```bash
npm run build && npm run preview
```

`npm run build` はfrontmatterのバリデーション・画像参照チェック・スラッグ重複チェックを兼ねているため、記事を追加したら**必ず一度実行する**こと。

## 8. 公開までの流れ

1. `main` を最新化してブランチを切る（`docs/post-<slug>` のような名前）
2. `content/posts/YYYY-MM-DD-<slug>/index.md` を作成（画像があれば同フォルダに追加）
3. `npm run dev` で見た目を確認
4. `npm run build` でビルドが通ることを確認
5. コミット（`docs: 「<タイトル>」の記事を追加`）してPRを作成（[CONTRIBUTING.md](CONTRIBUTING.md) 参照）
6. PRをマージすると、`main` への反映をトリガーに GitHub Actions（[.github/workflows/deploy.yml](../.github/workflows/deploy.yml)）が自動でビルド & GitHub Pages へデプロイする。数分後にサイトへ反映される
7. マージ後、ローカルの `main` を同期

> **ポイント**: 自動デプロイはビルドが成功したときだけ実行される。frontmatterの不備・画像の欠落・Mermaid構文エラーなどでビルドが失敗した場合はデプロイされず、既存の公開サイトはそのまま維持される。だからこそ、PRを出す前に手元で `npm run build` が通ることを必ず確認しておく。

## 9. よくあるビルドエラーと対処法

| エラーメッセージ（抜粋） | 原因 | 対処 |
|---|---|---|
| `frontmatterが不正です — title: Required` | `title`が未記入 | frontmatterに`title`を追加 |
| `frontmatterが不正です — date: Required` | `date`が未記入 | frontmatterに`date: YYYY-MM-DD`を追加 |
| `date "..." を日付として解釈できません` | `date`の値が日付として不正 | `2026-07-14`のようなISO形式で記入 |
| `参照先画像 "./xxx.png" が存在しません` | 画像パスのtypo、またはファイル未追加 | ファイル名・パスを確認、画像を同フォルダに配置 |
| `記事スラッグが重複しています: "foo" ← ...` | 日付を除いたスラッグが他記事と重複 | フォルダ名（スラッグ部分）を変更 |

## 10. 公開前チェックリスト

- [ ] フォルダ名が `YYYY-MM-DD-<slug>` 形式になっている
- [ ] `title` / `date` を記入した
- [ ] 一覧・OGPに表示したい `description` を書いた
- [ ] `draft: true` になっていない（公開する場合）
- [ ] 見出しは `##` から始めている（`#`を使っていない）
- [ ] `npm run build` がエラーなく通った
- [ ] `npm run dev` で表示・画像・コードハイライトを目視確認した
