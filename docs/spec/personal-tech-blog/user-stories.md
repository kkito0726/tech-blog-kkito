# 個人テックブログ ユーザストーリー

**作成日**: 2026-07-14
**関連要件定義**: [requirements.md](requirements.md)
**ヒアリング記録**: [interview-record.md](interview-record.md)

**【信頼性レベル凡例】**:
- 🔵 **青信号**: ユーザー依頼・ヒアリングを参考にした確実なストーリー
- 🟡 **黄信号**: ユーザー依頼・ヒアリングから妥当な推測によるストーリー

**登場ユーザー**:
- **執筆者**: ブログ所有者（本人）。記事を書いて公開する
- **読者**: ブログを訪問して記事を読む人

## エピック1: 記事の執筆と自動公開（Phase 1）

### ストーリー 1.1: マークダウンでの記事執筆 🔵

**信頼性**: 🔵 *ユーザー依頼・ヒアリングQ6より*

**私は** 執筆者 **として**
**記事フォルダを作成してfrontmatter付きマークダウンと画像をまとめて置きたい**
**そうすることで** 特別なCMSを使わず、使い慣れたエディタとGitだけで記事と画像を一体で管理できる

**関連要件**: REQ-001, REQ-002, REQ-011, REQ-012

**詳細シナリオ**:
1. 執筆者が `content/posts/2026-07-14-my-article/` フォルダを作成し、`index.md` を作成する
2. frontmatterに `title` と `date`（必須）、必要に応じて `description` と `draft` を記載する
3. 本文をマークダウンで執筆する。画像は同フォルダに置き、相対パス `![alt](./image.png)` で参照する
4. 公開後の記事URLは `/posts/my-article/` となる（日付はURLに含まれない）

**前提条件**:
- リポジトリをローカルにcloneしている

**制約事項**:
- `title` / `date` の欠落はビルドエラーになる（REQ-104）

**優先度**: Must Have

---

### ストーリー 1.2: pushによる自動公開 🔵

**信頼性**: 🔵 *ユーザー依頼「ファイルが増えるごとにGitHub Actionsで自動反映」より*

**私は** 執筆者 **として**
**記事ファイルをmainブランチにpushするだけでサイトに反映されるようにしたい**
**そうすることで** デプロイ作業を意識せず執筆に集中できる

**関連要件**: REQ-101, REQ-105, REQ-404, NFR-203

**詳細シナリオ**:
1. 執筆者が記事を追加・更新してmainブランチにpushする
2. GitHub Actionsが自動的にビルドを実行する
3. ビルド成功時、GitHub Pagesへ自動デプロイされ、数分以内に公開サイトに反映される
4. ビルド失敗時、ワークフローが失敗ステータスとなりデプロイは行われない

**前提条件**:
- GitHub Pages・GitHub Actionsの初期設定が完了している

**優先度**: Must Have

---

### ストーリー 1.3: 下書き記事の管理 🔵

**信頼性**: 🔵 *ヒアリングQ6より*

**私は** 執筆者 **として**
**`draft: true` を付けた書きかけの記事をリポジトリにpushしても公開されないようにしたい**
**そうすることで** 執筆途中の記事もバージョン管理しつつ、公開タイミングを制御できる

**関連要件**: REQ-102

**詳細シナリオ**:
1. 執筆者がfrontmatterに `draft: true` を記載してpushする
2. ビルドは成功するが、該当記事は一覧・詳細ページ・検索対象に含まれない
3. 公開時は `draft: true` を削除（または `false` に変更）してpushする

**優先度**: Must Have

## エピック2: 記事の閲覧体験（Phase 1）

### ストーリー 2.1: 記事一覧の閲覧 🟡

**信頼性**: 🟡 *ブログの一般的な構成からの妥当な推測*

**私は** 読者 **として**
**トップページで記事の一覧を新しい順に見たい**
**そうすることで** 最新の記事や興味のある記事をすぐに見つけられる

**関連要件**: REQ-003

**詳細シナリオ**:
1. 読者がブログのトップページにアクセスする
2. 記事のタイトル・公開日・説明文が公開日の降順で一覧表示される
3. 記事をクリックすると記事詳細ページへ遷移する

**優先度**: Must Have

---

### ストーリー 2.2: 記事詳細の閲覧 🔵

**信頼性**: 🔵 *ユーザー依頼・ヒアリングQ5より*

**私は** 読者 **として**
**シンタックスハイライトされたコードと目次付きで記事を読みたい**
**そうすることで** 技術記事のコードを読みやすく、長い記事でも目的のセクションにすぐ移動できる

**関連要件**: REQ-004, REQ-005, REQ-006

**詳細シナリオ**:
1. 読者が記事詳細ページにアクセスする
2. マークダウンがHTMLとしてレンダリングされ、コードブロックは言語に応じて色付けされる
3. 見出しから自動生成された目次が表示され、クリックで該当セクションにジャンプできる

**前提条件**:
- 記事にコードブロック・見出しが含まれる（含まれない場合、TOCは非表示: EDGE-103）

**優先度**: Must Have

---

### ストーリー 2.3: SNSシェアと検索エンジン流入 🔵

**信頼性**: 🔵 *ヒアリングQ3・Q5より*

**私は** 執筆者 **として**
**記事がSNSでシェアされた際にタイトル・説明文付きのカードで表示され、検索エンジンにも正しくインデックスされるようにしたい**
**そうすることで** ブログへの流入を増やせる

**関連要件**: REQ-007, REQ-008

**詳細シナリオ**:
1. 各記事ページがビルド時に静的HTML化され、OGP・SEOメタタグがHTMLに埋め込まれる
2. SNSのクローラーがメタタグを読み取り、シェア時にカードが表示される
3. 検索エンジンが各記事を個別ページとしてインデックスする

**優先度**: Must Have

## エピック3: サイト体験の拡張（Phase 2）

### ストーリー 3.1: 記事検索 🔵

**信頼性**: 🔵 *ヒアリングQ4より*

**私は** 読者 **として**
**キーワードで記事を検索したい**
**そうすることで** 記事が増えても目的の情報にすぐたどり着ける

**関連要件**: REQ-009, REQ-301

**詳細シナリオ**:
1. 読者が検索ボックスにキーワードを入力する
2. タイトル・本文にマッチする記事が一覧表示される
3. 該当なしの場合は空結果メッセージが表示される（EDGE-102）

**優先度**: Should Have

---

### ストーリー 3.2: ダークモード 🔵

**信頼性**: 🔵 *ヒアリングQ4より*

**私は** 読者 **として**
**ライト/ダークテーマを切り替えたい（初回はOS設定に従う）**
**そうすることで** 好みや環境に合った見た目で快適に記事を読める

**関連要件**: REQ-010, REQ-201, REQ-202

**詳細シナリオ**:
1. 初回訪問時、OSのカラースキーム設定に従ったテーマで表示される
2. 読者がヘッダーのトグルでテーマを切り替える
3. 選択したテーマはlocalStorage等に保存され、再訪問時も維持される

**優先度**: Should Have
