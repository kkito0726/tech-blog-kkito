# 個人テックブログ 要件定義書

**作成日**: 2026-07-14
**要件名**: personal-tech-blog

## 概要

マークダウンファイルとして記事を執筆し、リポジトリへのpushをトリガーにGitHub Actionsが自動でビルド・デプロイを行う個人テックブログを構築する。フロントエンドはVite + React + TypeScript + TailwindCSSで実装し、UIコンポーネントはAtomicデザインパターンに従って構成する。ビルド時に全ページを静的HTML化（SSG）し、GitHub Pagesで配信する。

**リリース方針**: 段階的リリース
- **Phase 1（コア）**: 記事一覧・詳細表示、SSG、GitHub Actions自動デプロイ、シンタックスハイライト、目次、OGP・SEOメタタグ、下書きフラグ
- **Phase 2（拡張）**: 記事検索、ダークモード

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: ユーザー依頼・ヒアリングを参考にした確実な要件
- 🟡 **黄信号**: ユーザー依頼・ヒアリングから妥当な推測による要件
- 🔴 **赤信号**: ユーザー依頼・ヒアリングにない推測による要件

### 通常要件

- REQ-001: システムは記事を「1記事1フォルダ」方式（コロケーション）で管理しなければならない。記事フォルダは `content/posts/YYYY-MM-DD-<スラッグ>/` 形式とし、本文はフォルダ直下の `index.md`（frontmatter付きマークダウン）とする 🔵 *ユーザー依頼・追加ヒアリングQ8より*
- REQ-002: システムはfrontmatterの必須項目として `title`（タイトル）と `date`（公開日）を、任意項目として `description`（説明文）と `draft`（下書きフラグ）をサポートしなければならない 🔵 *ヒアリングQ6より*
- REQ-003: システムはトップページに記事一覧を公開日の降順で表示しなければならない 🟡 *ブログの一般的な仕様からの妥当な推測*
- REQ-004: システムは記事詳細ページでマークダウンをHTMLにレンダリングして表示しなければならない 🔵 *ユーザー依頼より*
- REQ-005: システムは記事内のコードブロックにシンタックスハイライトを適用しなければならない 🔵 *ヒアリングQ5より*
- REQ-006: システムは記事の見出し（h2/h3）から目次（TOC）を自動生成して記事詳細ページに表示しなければならない 🔵 *ヒアリングQ5より*
- REQ-007: システムは各ページにtitle・description・OGP（og:title, og:description, og:url, og:type等）のメタタグを出力しなければならない 🔵 *ヒアリングQ5より*
- REQ-008: システムはビルド時に全ページを静的HTMLとして生成（SSG）しなければならない 🔵 *ヒアリングQ3より*
- REQ-009: システムは記事のキーワード検索機能（タイトル・本文対象）を提供しなければならない 🔵 *ヒアリングQ4より（Phase 2）*
- REQ-010: システムはライト/ダークテーマの切り替え機能を提供しなければならない 🔵 *ヒアリングQ4より（Phase 2）*
- REQ-011: 記事内で使用する画像は記事フォルダ内に配置し、本文からは相対パス（例: `![構成図](./architecture.png)`）で参照できなければならない。システムはビルド時に相対パスをハッシュ付きアセットURLへ解決しなければならない 🔵 *追加ヒアリングQ8より*
- REQ-012: 記事URLは `/posts/<スラッグ>/` 形式とし、スラッグはフォルダ名から日付プレフィックス（`YYYY-MM-DD-`）を除いた部分としなければならない（URLに日付は含めない） 🔵 *追加ヒアリングQ9より*

### 条件付き要件

- REQ-101: mainブランチに記事ファイルの追加・更新がpushされた場合、システムはGitHub Actionsで自動的にビルドを実行しGitHub Pagesへデプロイしなければならない 🔵 *ユーザー依頼・ヒアリングQ2より*
- REQ-102: frontmatterに `draft: true` が設定されている場合、システムはその記事を記事一覧・記事詳細ページ・検索対象・OGP出力のいずれにも含めてはならない 🔵 *ヒアリングQ6より*
- REQ-103: 存在しない記事URLへアクセスされた場合、システムは404ページを表示しなければならない 🟡 *静的サイトの一般仕様からの妥当な推測*
- REQ-104: frontmatterの必須項目（title, date）が欠落した記事ファイルが存在する場合、ビルドはエラーメッセージを出力して失敗しなければならない 🟡 *ユーザーのグローバルルール（入力バリデーション）からの妥当な推測*
- REQ-105: ビルドが失敗した場合、GitHub Actionsのワークフローは失敗ステータスを返し、デプロイを実行してはならない 🟡 *CI/CDの一般仕様からの妥当な推測*

### 状態要件

- REQ-201: ユーザーがテーマを切り替えた状態の場合、システムは再訪問時もそのテーマ設定を保持（localStorage等）しなければならない 🟡 *ダークモード採用からの妥当な推測*
- REQ-202: ユーザーが未選択の初回訪問状態の場合、システムはOSのカラースキーム設定（prefers-color-scheme）に従ってテーマを適用しなければならない 🟡 *ダークモード採用からの妥当な推測*

### オプション要件

- REQ-301: システムは検索機能をビルド時に生成した検索インデックスによるクライアントサイド検索として実装してもよい 🟡 *静的サイト構成からの妥当な推測*
- REQ-302: システムはfrontmatterにサムネイル画像等の追加メタデータ項目を将来拡張として追加できる構造にしてもよい 🟡 *段階的リリース方針からの妥当な推測*

### 制約要件

- REQ-401: システムはVite・React・TypeScript・TailwindCSSで実装しなければならない 🔵 *ユーザー依頼より*
- REQ-402: UIコンポーネントはAtomicデザインパターン（atoms / molecules / organisms / templates / pages）に従って構成しなければならない 🔵 *ユーザー依頼より*
- REQ-403: ホスティングにはGitHub Pagesを使用しなければならない 🔵 *ヒアリングQ2より*
- REQ-404: デプロイはGitHub Actionsのワークフローで自動化しなければならない 🔵 *ユーザー依頼より*
- REQ-405: GitHub Pagesのサブパス配信（`/<リポジトリ名>/`）に対応するため、Viteの `base` 設定およびルーティングのベースパスを設定しなければならない 🟡 *GitHub Pages仕様からの妥当な推測*
- REQ-406: タグ・カテゴリ分類、RSSフィード、コメント機能は本スコープに含めない（Won't Have） 🔵 *ヒアリングQ4・Q5で非選択*

## 非機能要件

### パフォーマンス

- NFR-001: 全ページを静的配信とし、記事詳細ページの初回表示（LCP）は一般的な回線で2.5秒以内であること 🟡 *SSG採用・Web Vitals基準からの妥当な推測*
- NFR-002: 記事検索は数百記事規模でも入力から1秒以内に結果を表示すること 🟡 *個人ブログの規模からの妥当な推測*
- NFR-003: Lighthouseのパフォーマンススコア90以上を目標とすること 🔴 *ヒアリングにない推測（目標値は要確認）*

### セキュリティ

- NFR-101: マークダウンのHTMLレンダリングにおいてXSSを防止すること（生HTMLを許可する場合はサニタイズを必須とする） 🟡 *ユーザーのグローバルセキュリティルールからの妥当な推測*
- NFR-102: リポジトリおよびビルド成果物にシークレット（APIキー等）を含めないこと 🟡 *ユーザーのグローバルセキュリティルールより*
- NFR-103: GitHub Actionsワークフローの権限（permissions）は必要最小限とすること 🟡 *CI/CDセキュリティの一般原則からの妥当な推測*

### ユーザビリティ

- NFR-201: モバイル・タブレット・デスクトップに対応したレスポンシブデザインとすること 🟡 *個人ブログの一般要件からの妥当な推測*
- NFR-202: ライト/ダーク両テーマにおいて本文・コードブロックの可読性を確保すること 🟡 *ダークモード採用からの妥当な推測*
- NFR-203: 記事執筆者（サイト所有者）はマークダウンファイルを追加してpushする以外の作業なしで記事を公開できること 🔵 *ユーザー依頼の中核要求*

### 保守性

- NFR-301: コンポーネントはAtomicデザインの階層に従い、1ファイル200〜400行を目安（最大800行）とすること 🔵 *ユーザーのグローバルコーディングルールより*
- NFR-302: テストカバレッジ80%以上を維持すること（単体・統合・E2E） 🔵 *ユーザーのグローバルテストルールより*

## Edgeケース

### エラー処理

- EDGE-001: frontmatterが不正（YAML構文エラー・必須項目欠落）な記事がある場合、ビルドは対象ファイル名を含むエラーを出力して失敗する 🟡 *REQ-104からの妥当な推測*
- EDGE-002: 日付プレフィックス除去後のスラッグ（URL）が重複した場合、ビルドは重複フォルダ名を含むエラーを出力して失敗する 🔵 *追加ヒアリングQ9より（URL形式選択時に確認済み）*
- EDGE-003: 存在しないURLへのアクセスは404ページへ誘導する 🟡 *REQ-103より*

### 境界値

- EDGE-101: 記事が0件の場合、記事一覧は空状態メッセージを表示し、ビルドは成功する 🟡 *新規ブログの初期状態からの妥当な推測*
- EDGE-102: 検索結果が0件の場合、「該当する記事がありません」等の空結果表示を行う 🟡 *検索機能の一般仕様からの妥当な推測*
- EDGE-103: 見出しが1つもない記事では目次（TOC）を表示しない 🟡 *TOC機能からの妥当な推測*
- EDGE-104: 同一公開日の記事が複数ある場合も一覧の並び順が安定していること（第2ソートキー: ファイル名等） 🔴 *ヒアリングにない推測（要確認）*
